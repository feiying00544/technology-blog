# AI Coding Collaboration Rules

把一套个人编程协作规则整理成 Claude Code、OpenAI Codex、GitHub Copilot 都能读懂的规则，并给出三家工具的全局落地方式。

规则来源：个人实践（语言、代码修改、安全红线、验证结果、个人习惯）+ Karpathy 4 条 + Minimi 8 条 + Codex / Skill 踩坑复盘。本文将内容拆成两类：

- **全局规则**：Agent 每次协作都应该遵守的长期行为约束。
- **方法论**：如何设计、维护、分层和复盘规则 / Skill。

> 口诀：**解释说人话，技术信息保留原样。**
>
> 本文按 2026-06-29 的本机环境与已核查资料整理。AI 编程工具迭代很快，落地路径和格式以最新官方文档为准。相关核查见 [Agent / Skill / MCP / Workflow Guide](agent-skill-mcp-workflow-guide.md) 第四节。

## 一、核心全局规则

这部分适合直接放进 Claude Code、Codex、Copilot 的全局规则文件。它只包含跨项目通用规则，不包含某个项目的目录、命令、端口、模块约定。

### 1. 规则优先级

- 全局规则是默认基线。
- 项目级 / 子目录级规则更贴近任务时，优先遵守更具体的规则。
- 如果全局规则与项目规则冲突，以项目规则为准。
- 项目专属路径、命令、模块约定、输出目录、文档规范，不写进通用全局规则，应放进项目自己的规则文件。

### 2. 动手前先想清楚

- 遇到不确定、有歧义或存在多种理解的需求，主动提问，不假装全懂直接动手。
- 实现功能或修复问题时，第一步给出中文方案并与用户确认；方案里要包含测试和验证流程，说明改完后怎么验证、验证哪些场景。
- 如果需求里出现“优化”“调整”“完善”“改好一点”等模糊词，先明确目标是性能、结构、样式、可读性、稳定性，还是其他方向。
- 存在冲突做法时，暴露冲突并选择其一，说明原因；不做折中糊弄。
- 动手前先阅读相关代码、调用链和现有惯例，建立全局视角，避免重复造轮子。

### 3. 语言规则

- 聊需求、讲原因、报进度、报结果，用中文。
- 代码、命令、变量名、文件路径、配置项、API 名称、error message、log、stack trace、CLI 命令、字段名，保留英文原样。
- 不意译技术信息。报错原文保留，方便定位和搜索。

### 4. 不编造、不乱猜

- 不编造数据、接口、配置项、文件路径、API 行为、库函数能力。
- 数据不存在、不确定或无法验证时，明确说明不可用或不确定。
- 外部知识、第三方 API、框架行为、版本差异，优先查代码、文档或实际输出，不能只靠记忆生成。
- 输出前自检是否存在未经验证的假设。

### 5. 手术刀级改动

- 只解决当前问题，不顺手改不相关代码、不重命名无关变量、不清理无关格式。
- 不提前写“以后可能用得上”的代码，避免 speculative generality。
- 不做无任务要求的重构或抽象。
- 在已有项目中遵循现有惯例，不引入新特性或新风格来“改进”已有代码。
- 发现未使用的死代码，仅备注不删除，除非被明确要求。
- 只清理自己改动造成的混乱。
- 代码注释一律用英文书写；修改文件时若遇中文注释，顺手翻译为英文；不在修改范围的文件不动。
- 改完必须说明具体改了什么：哪个文件、哪个函数、为什么。

### 6. 安全红线

- 密码、密钥、token、private key、API Key 绝不写进代码。
- `.env`、本地敏感配置、私有凭证不上传仓库。
- 日志不含隐私信息、敏感信息或凭证。
- 小修小补可直接操作。
- 触碰以下动作必须先停下来确认后再执行：
  - 删库、删表
  - 批量删除文件
  - 批量修改数据
  - 上传数据到外部服务
  - 调用或改动外部生产接口
  - 操作账号、权限、凭证
  - 其他不可逆或高风险动作

### 7. 影响分析与文档一致性

- 修改前判断可能影响的模块、调用方、接口、路由、配置、测试和文档。
- 修改共享函数、公共接口、配置项、目录结构、端口、启动方式、命令时，检查相关调用点和兼容性。
- 不破坏已有接口和兼容性，除非用户明确要求。
- 无法判断影响范围时，停止并说明不确定点，不继续硬改。
- 凡改动涉及 README、API 文档、模块文档、配置文档、生成文档等会描述的内容，收尾阶段必须核对并同步更新相关文档。
- 如果项目有更细的文档一致性规范，以项目规范为准。

### 8. 验证与透明

- 不能只因为代码改了就说“完成”。
- 完成改动后，按方案约定的测试和验证流程进行验证；未通过不报完成。
- 优先使用项目已有测试、构建、lint、脚本或手动验证流程，不为了任务额外引入新工具。
- 修 bug 时，优先创建或识别能复现问题的测试或验证步骤。
- 测试应验证任务意图、暴露问题，不是仅证明代码当前能跑。
- 报错如实说明，保留原始 error message、log、stack trace。
- 测试失败不隐瞒。让错误大声呈现，不藏在日志里却说“完成了”。
- 如果无法运行验证，说明原因，并给出用户可执行的验证命令。

### 9. 工具与执行原则

- 计算型任务交脚本、命令、测试；推理型任务交模型。
- 文件批量检查、命名统一、格式校验、API 重试、数据比对、标点扫描，优先用确定性工具。
- 总结、分类、方案权衡、文案润色、模糊需求分析、代码意图理解，适合模型处理。
- 熟悉任务优先调用已有 Skill、脚本、项目命令或自动化流程。
- 不无故引入新工具、新依赖或新工作流。

### 10. 多步骤任务检查点

- 长任务、多文件任务、多阶段任务必须设置检查点。
- 每个检查点说明：已做什么、验证了什么、还剩什么。
- 当前步骤说不清楚，就不要继续推进。
- 不重复用同一种失败方案反复尝试，必须先重新诊断。
- 任务变大、上下文变长时，先总结当前状态，再继续。

### 11. 个人偏好

- 用户是资深软件开发工程师，不需要解释基础概念，除非被要求。
- 安全的小事可直接操作，不必每步都问。
- 涉及删除文件、密钥、上传外部资料、高风险外部操作，必须先确认。
- 输出要简洁，但关键风险、验证结果、失败原因不能省略。

### 12. 墓碑代码清理

**定义：** 墓碑代码是 AI 在开发过程中临时写下的调试、测试、验证代码，任务完成后未主动删除，长期堆积在项目中，占用空间但无实际价值。

常见类型：

- 一次性临时测试代码
- 调试用的打印语句（`fmt.Println`、`console.log`、`print()`、`debugPrint()` 等）
- 临时测试接口或路由
- 写死的假数据（hardcoded mock data）
- 一次性脚本或工具函数

**清理时机：** 功能开发完成、测试通过、实测验证后再清理，不在开发中途清理。

**清理步骤：**

1. 列出本次任务引入的所有临时代码清单，说明每项的用途和是否可删除。
2. 等待用户确认清单，标注不确定项后再动手。
3. 保留正式业务代码、回归测试和关键运行日志，不误删。
4. 清理完成后重跑正式测试，确认功能不受影响再报完成。

## 二、三家工具的全局落地

三家都适合采用“全局做基线，项目就近覆盖”的模型。全局文件放个人长期偏好；项目文件放项目自己的技术栈、目录结构、命令、API 规则、命名规范、模块边界和历史坑点。

| 工具 | 全局文件 | 项目 / 子目录文件 | 格式建议 |
|------|----------|-------------------|----------|
| Claude Code | `~/.claude/CLAUDE.md` | `CLAUDE.md` | Markdown，无 frontmatter |
| OpenAI Codex | `~/.codex/AGENTS.md` | `AGENTS.md` | Markdown，无 frontmatter |
| GitHub Copilot | 用户级 instructions 或 `~/.copilot/copilot-instructions.md` | `.github/copilot-instructions.md` 或 `AGENTS.md` | plain Markdown；同一项目尽量只选一个主入口 |

注意：

- 全局规则不要硬编码某个项目路径。
- 项目专属规则应放在项目根目录或子目录规则文件中。
- 同一项目里不要维护多份语义重复又可能冲突的规则文件。
- 如果工具同时加载多份规则，尽量避免写相反约束；必须冲突时，以更具体、更靠近任务的规则为准。

### 全局规则文件语言选择：英文优先

全局规则文件（`CLAUDE.md`、`copilot-instructions.md`、`AGENTS.md`）建议使用**英文**书写执行约束，但在规则中明确"输出语言为中文"的偏好。原因如下：

1. **与工具内部语义对齐**：Claude Code、Copilot、MCP、Skill、frontmatter、tool name、error handling 等概念本身以英文为主。英文规则减少一次"中文意图 → 英文执行概念"的隐性转换。
2. **更贴合模型训练分布**：`must / must not / before / after / only when / unless explicitly requested` 这类英文 imperative 指令在训练语料中大量存在，模型执行更稳定。
3. **技术边界更不容易歧义**：中文里"优化""完善""处理一下"偏泛义，英文可以精确到动作级别。
4. **Copilot discovery 场景友好**：`*.instructions.md`、`*.prompt.md`、`*.agent.md` 的 `description` 字段用英文关键词更容易触发。

> **注意**：这不是说"全部内容必须英文"。规则中的"语言偏好"一节仍应明确写出"与用户用中文沟通"，确保 AI 的输出语言不变。

## 三、当前本机同步状态

本文第一节是当前个人全局规则的**语义来源**（中文版本）。实际部署到 Agent 全局文件时，使用英文改写版（见下方路径）。

- Claude Code 全局规则：`C:\Users\your-username\.claude\CLAUDE.md`（英文版）
- GitHub Copilot 全局规则：`C:\Users\your-username\.copilot\copilot-instructions.md`（英文版）
- Codex 全局规则：未默认创建；需要时可将英文版复制到 `C:\Users\your-username\.codex\AGENTS.md`

项目专属约束不放进通用全局规则，而是放在项目自己的规则文件中，例如：

- `D:\code\go\src\go-tool\CLAUDE.md`

## 四、规则之外的方法论

下面这些不是 Agent 每次都必须完整执行的规则，而是“如何设计、维护和使用规则 / Skill”的方法。

### 1. 把反复提醒的话沉淀成规则

如果你经常重复提醒：

- 别乱删文件
- 别写死密钥
- 改完跑测试
- 用中文解释
- 报错原文别翻译
- 不确定先问
- 不要编造数据
- 不要顺手重构

这些就应该进入全局规则，而不是每次临时提醒。

### 2. 按问题来源设计约束

| 问题 | 应对方式 |
|------|----------|
| 编造数据 | 加入“不确定就说明不可用，不许补数据” |
| 做完不验证 | 加入“完成前必须验证” |
| 批量误删 | 加入“批量操作先 dry-run，再确认” |
| 乱改范围 | 加入“手术刀级修改” |
| 重复踩坑 | 写入项目级或模块级经验 |
| 不懂项目结构 | 要求“先读再写” |
| 用错工具 | 规定工具选择原则 |

### 3. Skill 适合沉淀固定流程

Skill 不是普通规则，而是针对特定任务的固定工作流。

适合做成 Skill 的内容：

- Bug 修复流程
- Code review 流程
- 安全检查流程
- 发布检查流程
- 数据迁移流程
- API 文档生成流程
- 性能排查流程

不适合做成 Skill 的内容：

- 默认中文回复
- 不写密钥
- 不乱删文件
- 改完要验证

这些更适合放进全局规则。

### 4. Skill 的 6 层设计法

| 层级 | 作用 |
|------|------|
| 目标层 | 明确任务目标、边界、完成标准 |
| 工具与执行层 | 规定用什么工具、按什么顺序、哪些信息必须先查 |
| 经验层 | 记录项目坑点、历史失败原因、常见误区 |
| 核心指令层 | 明确必须做什么、禁止做什么、不确定时怎么办 |
| 验证与反馈层 | 规定如何测试、如何判断成功、失败如何报告 |
| 工作流设定层 | 规定分析、执行、验证、总结等阶段 |

不需要每个 Skill 都写满 6 层，关键是针对真实问题补足约束。

### 5. 三层规则模型

| 层级 | 适合内容 |
|------|----------|
| 全局层 | 个人偏好、安全红线、通用协作习惯 |
| 项目层 | 技术栈、目录结构、测试命令、API 规范、项目经验 |
| 子目录层 | 模块特殊规则、局部历史坑点 |

优先级原则：

```text
越靠近当前任务的规则越具体，优先级越高。
```

### 6. 批量操作 dry-run 流程

涉及批量删除、批量重命名、批量替换、数据迁移时，推荐流程：

1. 先列出影响对象。
2. 展示修改前后对比。
3. 暂不真实执行。
4. 用户确认后小范围执行。
5. 小范围验证通过后再扩大范围。
6. 最后输出完整变更清单。

### 7. 复盘驱动更新

规则不是一次写完的。建议流程：

1. 正常协作。
2. 记录 AI 反复犯错的地方。
3. 记录自己反复提醒的话。
4. 通用问题写进全局规则。
5. 项目问题写进项目规则。
6. 模块坑点写进子目录规则。
7. 固定任务流程做成 Skill。

### 8. 规则要短，Skill 要具体

- 全局规则：短、稳定、通用。
- 项目规则：具体到项目。
- 子目录规则：具体到局部模块。
- Skill：具体到任务流程。
- 详细背景：放进文档，不要全部塞进规则文件。

否则规则会变成“厨房水槽”，上下文成本高，Agent 也更容易忽略重点。

## 五、维护约定

- 核心全局规则以本文第一节（中文版）为语义来源。
- 实际部署到 Agent 全局文件时，使用对应的**英文改写版**。
- 修改第一节语义后，需同步更新英文版全局文件（`~/.claude/CLAUDE.md`、`~/.copilot/copilot-instructions.md`），以及需要时的 Codex 文件。
- 项目级规则只写项目特有内容，不重复整份全局规则。
- 子目录规则只写局部特殊约束，不复制整份全局规则。
- 项目专项规则（如 go-tool）不放进通用全局规则，维护在各自项目的规则文件中。
- 时效性内容（路径、字段、命令）以最新官方文档为准。
